# 01 架构与并发模型

## 分层

```
SwiftUI Views(Features/*)
    ↓ 观察 @Observable
ViewModels(ScanViewModel / DeviceDetailViewModel / …)
    ↓ 只依赖协议
BLECentral 协议(Core/BLE/BLECentral.swift)
    ↙                    ↘
CentralManager        MockCentral
(CoreBluetooth)       (虚拟设备)
    ↓
PeripheralSession(单设备 GATT 会话)
```

核心原则:**业务与 UI 只依赖 `BLECentral` 协议,不直接触碰 CBPeripheral/CBCentralManager**。收益:

1. 模拟器(无蓝牙硬件)可用 `MockCentral` 走通全流程,开发调试不依赖真机+真设备;
2. 可注入故障(连接超时、强制断连)做异常路径演示与测试;
3. 换底层实现(如接厂商 SDK)不动上层。

数据模型(`DiscoveredDevice`/`GATTService`/`GATTCharacteristic`)同样与 CoreBluetooth 解耦,只在字段级引用 `CBUUID` 这类纯值语义类型。

## 并发模型:为什么全 MainActor

工程开启了 Xcode 26 模板默认的 `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`(approachable concurrency):所有类型默认主执行体隔离。配合 `CBCentralManager(delegate: self, queue: nil)`(回调派发主队列),得到一个**零锁**的并发模型:

- 委托回调、continuation 表读写、@Observable 状态更新全部发生在主线程,没有数据竞争;
- 委托遵循用 `@preconcurrency CBCentralManagerDelegate`:委托协议本身是非隔离的,`@preconcurrency` 让 MainActor 隔离的方法满足协议要求,运行时动态校验回调确实落在主执行体上(主队列即主执行体,成立)。

**高吞吐场景的改法**(如 OTA 传输、每秒数百帧):委托队列换专用串行队列 `CBCentralManager(delegate: self, queue: bleQueue)`,封装层改为 `actor` 或加锁,只在状态出口跳回 MainActor 刷 UI。本项目为教学清晰选择了前者,切换点集中在 `CentralManager.init` 与各类型的隔离标注。

## 回调 → async/await 的三种封装模式

CoreBluetooth 是"发起调用 + 委托回调"的分离式 API,封装规则按回调次数分三类:

### 1. 一次性回调 → CheckedContinuation

读、写、RSSI、服务发现。见 [PeripheralSession.swift](../WBBlueSwift/Core/BLE/PeripheralSession.swift):

```swift
func readValue(characteristic uuid: CBUUID) async throws -> Data {
    let characteristic = try findCharacteristic(uuid)
    return try await withCheckedThrowingContinuation { continuation in
        readContinuations[uuid] = continuation      // 按特征 UUID 键控挂起
        peripheral.readValue(for: characteristic)
    }
}
// 委托回调里:
// readContinuations.removeValue(forKey: uuid)?.resume(returning: data)
```

**三条铁律**:

- **恰好 resume 一次**。全部用 `removeValue(forKey:)?.resume(...)` 模式:取出即删除,天然防双 resume;
- **必须有失败出口**。断连时 `failAll(_:)` 把所有挂起 continuation 以 `BLEError.disconnected` 收尾,否则 await 方**永久悬挂**——这是 continuation 封装最经典的泄漏事故;
- **超时自己做**。CoreBluetooth 的 `connect` 永不超时,`CentralManager.connect` 用一个并行的超时 Task 竞速,超时方负责 `cancelPeripheralConnection`(不撤销的话设备回到范围会"幽灵连接"成功)。

### 2. 多值回调 → AsyncStream / AsyncThrowingStream

扫描结果、通知数据、连接事件、状态变化。

```swift
let (stream, continuation) = AsyncThrowingStream<Data, Error>.makeStream()
notifyStreams[uuid] = continuation
continuation.onTermination = { _ in /* 消费方取消 → setNotifyValue(false) 关 CCCD */ }
return stream
```

生命周期用 SwiftUI 的 `.task {}` 驱动:视图消失 → 任务取消 → 流终止 → `onTermination` 自动关通知。资源管理不需要手动配对 open/close。

### 3. 状态机 → @Observable + 事件流消费

`ReconnectOrchestrator` 消费 `connectionEvents` 流,把断连/重试/放弃收敛为一个 `phase` 属性供 UI 渲染。

## 依赖注入与数据源切换

`AppModel`(依赖容器)持有 `mockCentral` 与懒创建的 `realCentral`——**真实 CentralManager 一经创建即触发系统蓝牙权限弹窗**,所以不用就不创建(权限最佳实践)。切换数据源时 `generation += 1`,根视图 `.id(app.generation)` 整树重建,确保旧 central 的所有流被干净取消。
