# 03 异常处理手册

企业 BLE 开发的真正难点不在"连上",而在"各种连不上、断了、卡住"。本篇按场景逐条给出:现象 → 原因 → 方案 → 本项目代码落点。统一错误模型见 [BLEError.swift](../WBBlueSwift/Core/BLE/BLEError.swift)(每个错误都带 `recoverySuggestion`,UI 直接展示)。

Mock 数据源下可用 扫描页 ⚙️ → 故障注入 现场复现 №3、№4。

## 1. 蓝牙关闭 / 未授权 / 不支持

- **现象**:什么都扫不到;`CBCentralManager.state` 不是 `.poweredOn`。
- **原因**:用户关了蓝牙(poweredOff)、拒绝了权限(unauthorized)、模拟器或老设备(unsupported)。
- **方案**:
  - 任何 API 调用前置检查 `state`,非 poweredOn 直接抛对应 `BLEError`(`CentralState.asError`);
  - UI 常驻状态横幅,unauthorized 时给"打开设置"按钮跳 `UIApplication.openSettingsURLString`;
  - 订阅 `stateStream()` 驱动 UI,状态恢复 poweredOn 后自动允许重新操作。
- **代码**:`ScanView.StateBanner`、`CentralManager.centralManagerDidUpdateState`。
- **坑**:`poweredOff/resetting` 时系统隐式断开所有连接且**不回调 didDisconnect**,`CentralManager` 在状态回调里手动 `failAll` + 广播断连事件,否则上层还以为连着。

## 2. 扫描不到设备

- **原因优先级**:设备没在广播(被别的手机连走了——BLE 外设建连后通常停止广播)> 服务过滤 UUID 不对 > 设备广播间隔太长 > 距离/遮挡。
- **方案**:扫描超时(20s)自停并提示;提供"关闭过滤"开关排查;RSSI 排序帮用户找近处设备。
- **代码**:`ScanViewModel.startScan`(超时/过滤/幽灵清理三件套)。

## 3. 连接超时(演示:故障注入"连接一直超时")

- **现象**:`connect` 后既不 `didConnect` 也不 `didFailToConnect`——CoreBluetooth 的连接请求**永不超时**。
- **方案**:自己实现超时竞速;超时后**必须** `cancelPeripheralConnection`,否则连接意图一直挂在系统里,设备哪天出现就"幽灵连接"成功,App 状态错乱。
- **代码**:`CentralManager.connect(id:timeout:)` 的 timeoutTask + `failConnect`。

```swift
let timeoutTask = Task {
    try? await Task.sleep(for: .seconds(timeout))
    guard !Task.isCancelled else { return }
    self?.failConnect(id: id, error: BLEError.timeout(operation: "连接"))  // 内部 cancelPeripheralConnection
}
defer { timeoutTask.cancel() }
try await withCheckedThrowingContinuation { ... central.connect(peripheral) }
```

## 4. 意外断连与自动重连(演示:故障注入"15 秒强制断连")

- **现象**:`didDisconnectPeripheral` 带 error(超距、设备关机/复位、链路 supervision timeout)。
- **方案**:三层配合——
  1. **区分主动/被动**:主动断开(error == nil)绝不重连;`CentralManager` 用 `intentionalDisconnects` 集合标记;
  2. **指数退避 + 抖动**:立即重试会加剧射频拥塞;`Backoff.delay`(1s 起步、2 倍增长、30s 封顶、50% 抖动),重试 6 次放弃并明确告知用户;
  3. **重连后重建现场**:GATT 句柄与 CCCD 订阅在断连后**全部失效**,重连成功必须重新 discoverServices + 重新订阅——这是最常被漏掉的坑。
- **代码**:[ReconnectOrchestrator.swift](../WBBlueSwift/Core/BLE/ReconnectOrchestrator.swift)(状态机),`DeviceDetailViewModel.onConnected`(重建订阅:保留"订阅意图"集合 `notifyingCharacteristics`,重连后按意图恢复)。

## 5. 挂起的 async 调用因断连永久悬挂

- **现象**:`await readValue(...)` 永远不返回,上层 Task 泄漏,UI 转圈。
- **原因**:continuation 封装只在成功回调里 resume,断连后委托回调不再来。
- **方案**:会话级 `failAll(_:)`——断连时把**所有**挂起的读/写/发现/RSSI continuation 以 `BLEError.disconnected` resume,所有通知流 `finish(throwing:)`。
- **代码**:`PeripheralSession.failAll`,由 `CentralManager.didDisconnectPeripheral` 调用。**写 continuation 封装必须同步写失败出口,这是纪律。**

## 6. 免响应写丢数据(背压)

- **现象**:`writeValue(.withoutResponse)` 连发大量数据,对端只收到一部分,不报任何错。
- **原因**:发送缓冲满后系统**静默丢弃**。
- **方案**:每次写前检查 `peripheral.canSendWriteWithoutResponse`,false 则等待(正式项目用 `peripheralIsReady(toSendWriteWithoutResponse:)` 回调唤醒,本项目用轮询简化);外设侧同理,`updateValue` 返回 false 要等 `peripheralManagerIsReady`。
- **代码**:`PeripheralSession.writeValue` 的等待循环;`PeripheralModeViewModel.pushHeartRate` 的 `waitingForQueue`。

## 7. 订阅失败:insufficientAuthentication

- **现象**:`didUpdateNotificationState` 带 `CBATTError.insufficientAuthentication`(错误码 5);读写也可能报同样错误。
- **原因**:特征权限要求加密链路(`.notifyEncryptionRequired` 等),而当前未配对。
- **方案**:iOS 会自动弹配对框,App 侧把错误映射为人话("该特征要求加密链路:接受配对后重试"),用户接受配对后重试即可。详见 [06-配对绑定与安全](06-配对绑定与安全.md)。
- **代码**:`BLEError.att(.insufficientAuthentication)` 的 `recoverySuggestion`。

## 8. 协议层坏帧:半包、粘包、CRC 错误

- **现象**:通知回调到达的 Data 不是一帧完整业务数据:可能只有半帧(按 MTU 切割)、可能两帧粘连(固件缓冲合并)、可能被链路误码污染。
- **方案**:流式组包状态机——缓冲累积、找帧头、按 len 等整帧、CRC 校验;坏帧丢弃后**从帧头之后逐字节重同步**,一个坏帧不拖垮整条流;len 字段设上限(512B)防损坏数据撑爆内存。
- **代码**:[PacketCodec.swift](../WBBlueSwift/Core/Protocol/PacketCodec.swift) `PacketAssembler`,11 项单测覆盖全部路径(`WBBlueSwiftTests/PacketCodecTests.swift`)。

## 9. 重复操作与竞态

- **场景**:用户连点"连接"、同一特征重复订阅、扫描中再扫描。
- **方案**:幂等化或显式拒绝——`connect` 检查已有挂起 continuation 则抛"正在连接中";`startScan` 自动替换旧流;重复订阅 finish 旧流再建新流;`DiscoveredDevice` 按 id upsert 去重。
- **代码**:`CentralManager.connect` 前置检查、`PeripheralSession.notifications` 的替换逻辑、`ScanViewModel.upsert`。

## 10. App 后台被杀后连接丢失

- **现象**:App 退后台一段时间被系统回收,设备数据中断。
- **方案**:`bluetooth-central` 后台模式 + State Restoration(系统代持连接,事件到达时拉活 App)。工程配置与代码模式见 [05-后台模式与状态恢复](05-后台模式与状态恢复.md)(示例工程为保持模板简洁未开启,文档给出完整接入步骤)。

## 排障总入口:日志
