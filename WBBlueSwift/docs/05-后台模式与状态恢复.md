# 05 后台模式与状态恢复

> 本示例工程为保持模板简洁**未开启**后台能力(所有演示前台即可完成);本篇给出企业项目的完整接入步骤与行为差异,照做即可。

## 三种运行形态的蓝牙能力

| 形态 | 扫描 | 连接/数据 | 广播(外设角色) |
|---|---|---|---|
| 前台 | 完整 | 完整 | 完整 |
| 后台(开了 bluetooth-central) | 只允许**带服务过滤**的扫描,扫描间隔被拉长,AllowDuplicates 失效 | 保持连接、正常收通知,事件唤醒 App 约 10s | 降级(见下) |
| 被系统杀死 | ✗ | 靠 State Restoration 复活 | ✗ |

## 第一步:开启后台模式

`Signing & Capabilities` → `Background Modes` → 勾选:

- `Uses Bluetooth LE accessories`(bluetooth-central,App 作为中心)
- 需要后台继续广播才勾 `Acts as a Bluetooth LE accessory`(bluetooth-peripheral)

等价于 Info.plist:

```xml
<key>UIBackgroundModes</key>
<array>
    <string>bluetooth-central</string>
</array>
```

开启后:App 退后台仍保持连接;设备通知到达时系统唤醒 App(约 10 秒处理窗口);后台扫描必须带服务 UUID 过滤(本工程扫描层已按此设计,见 `BLECentral.startScan(services:)` 注释)。

## 第二步:State Restoration(被杀后复活)

系统在 App 被回收后**代持蓝牙会话**;目标设备有事件(连接成功、通知到达)时,系统在后台重启 App 并归还会话。接入三件事:

```swift
// 1. 创建 manager 时给恢复标识(CentralManager.init 中)
central = CBCentralManager(
    delegate: self, queue: nil,
    options: [CBCentralManagerOptionRestoreIdentifierKey: "com.wb.WBBlueSwift.central"]
)

// 2. 实现恢复回调:系统把代持的外设还给你
func centralManager(_ central: CBCentralManager,
                    willRestoreState dict: [String: Any]) {
    if let peripherals = dict[CBCentralManagerRestoredStatePeripheralsKey] as? [CBPeripheral] {
        for peripheral in peripherals {
            // 重新持有 + 重设 delegate(否则回调丢失)
            knownPeripherals[peripheral.identifier] = peripheral
            sessions[peripheral.identifier] = PeripheralSession(peripheral: peripheral)
        }
    }
}

// 3. AppDelegate 里检查启动原因(SwiftUI 用 UIApplicationDelegateAdaptor)
// launchOptions[.bluetoothCentrals] 非空 = 因蓝牙事件被拉活,
// 此时必须用相同 restore identifier 重建 CBCentralManager。
```

**坑清单**:

- willRestoreState 在 `centralManagerDidUpdateState` **之前**回调;
- 恢复回来的 CBPeripheral 必须立刻强持有并重设 delegate;
- 用户手动杀 App(上滑)**不会**恢复——这是系统政策,不是 bug;
- 模拟器不支持 State Restoration,必须真机验证。

## 长连接保活的现实策略

iOS 没有"前台服务"(对比 Android),长连接产品(手环、血糖仪)的标准组合:

1. bluetooth-central 后台模式:退后台不断连;
2. State Restoration:被回收后靠设备事件拉活;
3. 固件配合:设备周期性发通知(心跳),既是业务数据也是拉活信号;
4. `CBConnectPeripheralOptionEnableTransportBridgingKey` 等选项按需;
5. 接受现实:用户手动杀 App 后无解,产品设计上引导用户不要杀。

## 后台广播降级(外设角色,真机可复现)

App 作为外设退到后台后:

- 广播包里 **LocalName 丢失**;
- 服务 UUID 从主广播区移到苹果私有的 **overflow 区**——只有 iOS 设备"定向扫描该服务"时才能发现,**多数安卓设备扫不到**;
- 广播频率降低。

本项目"外设模式"页 footer 有此提示;用一台安卓装 nRF Connect 对着退后台的 iPhone 扫一次,印象会非常深刻。相关:[02-CoreBluetooth 核心知识](02-CoreBluetooth核心知识.md) 的广播包结构。
