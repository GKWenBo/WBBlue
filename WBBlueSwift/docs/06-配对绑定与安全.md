# 06 配对绑定与安全

## 概念区分(面试高频)

- **配对(Pairing)**:一次性的密钥协商过程,建立加密链路;
- **绑定(Bonding)**:把配对产生的长期密钥(LTK)存下来,下次连接直接加密,不用重新配对;
- **白名单/自动重连**:绑定之后链路层可以只接受已绑定设备。

## 配对方式(由双方 IO 能力协商决定)

| 方式 | 交互 | 防中间人(MITM) |
|---|---|---|
| Just Works | 无(直接弹"是否配对"确认框) | ✗ |
| Passkey Entry | 一端显示 6 位数字,另一端输入 | ✓ |
| Numeric Comparison(LESC) | 两端显示同一数字,确认一致 | ✓ |
| OOB(带外) | NFC 等外部信道交换密钥 | ✓ |

BLE 4.2 起的 LE Secure Connections(LESC)用 ECDH 密钥交换,安全性显著高于 Legacy Pairing。

## iOS 的现实:没有配对/绑定 API

CoreBluetooth **没有**主动发起配对、查询绑定状态、解除绑定的 API(对比 Android 的 `createBond`)。配对完全由系统按需触发:

1. App 访问一个**要求加密**的特征(读/写/订阅);
2. 外设回 ATT 错误 `Insufficient Authentication (5)` / `Insufficient Encryption (15)`;
3. iOS 自动发起配对,弹系统配对框;
4. 用户接受 → 链路加密 → 系统**自动重试**刚才的操作,成功回调照常到达;用户拒绝 → 错误原样抛给 App。

**因此 App 侧的正确姿势**(本项目实现):

- 把 `CBATTError.insufficientAuthentication` 映射为可读提示("该特征要求加密链路:接受配对后重试"),见 [BLEError.swift](../WBBlueSwift/Core/BLE/BLEError.swift) `recoverySuggestion`;
- 订阅失败路径覆盖:`PeripheralSession.didUpdateNotificationState` 会把该错误抛给 `notifications()` 的调用方(见 [03-异常处理手册](03-异常处理手册.md) №7);
- 解除绑定只能引导用户:设置 → 蓝牙 → 设备 → 忽略此设备。**外设侧删除了绑定而 iPhone 还留着**(或反之)会造成"配对信息不一致,永远连不上",客服高频问题,解法就是两边都删重配。

## 外设固件侧如何要求加密

定义特征时给加密属性/权限(本项目外设模式演示的是开放特征;要求加密的写法):

```swift
CBMutableCharacteristic(
    type: uuid,
    properties: [.read, .notifyEncryptionRequired],
    value: nil,
    permissions: [.readEncryptionRequired]
)
```

## 隐私:随机地址与 identifier

- BLE 设备普遍使用**可解析随机地址(RPA)**,周期性更换,防跟踪;
- iOS 对 App 隐藏 MAC 地址,只给本机会话级 `peripheral.identifier`(UUID);绑定后系统内部用 IRK 解析 RPA,App 无感;
- 企业设备绑定用户账号时,**不要**依赖 identifier 做持久主键,应连接后读序列号(设备信息服务 0x180A)或厂商数据,参见 `DiscoveredDevice.id` 的注释。

## 传输层之上的应用层安全

很多企业设备不走 GATT 加密(跨平台行为不一致、固件实现复杂),而在**私有协议层**自建安全:挑战-响应认证(连接后 App 用预共享密钥/服务器签发的 token 应答设备挑战)、payload AES 加密、防重放(帧内嵌计数器)。这层就构建在 [04-私有二进制协议](04-私有二进制协议.md) 的帧结构之上(payload 换成密文即可),门锁类产品几乎必做。
