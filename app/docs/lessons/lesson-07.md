# 第 7 课：稳定性工程 · 自动重连（1 课时）

> 前 6 课让功能「跑通」，这课让它「跑稳」。企业蓝牙 App 和玩具 Demo 的分水岭，就是断线后能不能自己恢复。本课实现一个带指数退避的自动重连状态机，并处理它牵出的一连串坑。

## 一、断线分两种，只有一种该重连

第 3 课我们知道断线只能靠 `connectionState` 流感知，靠 `disconnectReason` 拿原因。本课的第一条设计决策：**区分「用户主动断开」和「意外掉线」，只对后者重连**。

- 用户在 UI 上点了「断开」→ 绝不能重连，否则用户断不掉，体验灾难。
- 设备走远、关机、被系统踢 → 应该自动重连，这才是稳定性价值所在。

代码里用一个 `_userInitiated` 标志区分：`disconnect()` 里置 `true`，`connect()` 里置 `false`。断线事件到达时先看这个标志（[device_controller.dart](../../lib/features/device/device_controller.dart)）。这对应 Swift 版 `ReconnectOrchestrator` 里「`error == nil` 就停在 idle」的判断——两端同一个思想。

> 进阶：也可以进一步用 `disconnectReason` 的 code 细分——安卓 status 8（超时）值得重连，status 19（对端主动断）可能不该重连。课程先用「用户意图」这个最可靠的信号。

## 二、为什么必须指数退避 + 抖动

掉线后立即重连、失败再立即重连，是新手写法，有三个害处：

1. **加剧射频拥塞**：连接失败往往就是因为信号差/干扰，狂发连接请求只会让射频环境更糟。
2. **耗电**：连接尝试是高功耗操作，无节制重试能把电池烧穿。
3. **惊群效应**：一个网关下几十个设备同时掉线（如网关重启），若都立即重连会在同一毫秒挤爆信道。

解法是**指数退避**：第 n 次重试前等 `base × 2^(n-1)` 秒，封顶 cap。再叠加**抖动（jitter）**：在等待时间上乘一个随机因子，把「同时掉线的一批设备」的重连时刻打散开。实现见 [backoff.dart](../../lib/core/backoff.dart)，纯函数、随机源可注入，4 条单测断言了「1→2→4→8 封顶 30」和抖动区间。

我们的参数：base 1s、cap 30s、最多 6 次，退避序列约 `1,2,4,8,16,30`（各带抖动），累计约 1 分钟后放弃。放弃后进入 `failed` 态，交给用户手动重连——**永不无限重试**，这也是省电与体验的平衡。

## 三、重连状态机

`DeviceController` 里新增 `ReconnectPhase`：`idle → connecting → connected`，掉线后 `waitingRetry(等待中) → connecting(尝试) →` 成功回 `connected` 或失败累加到 `failed`。UI 状态卡片直接显示 `reconnectText`，你能在真机上看到「断线，4.0s 后第 3 次重连」这样的实时文案。

关键实现点：

- **先监听后连接**：`connectionState` 订阅在构造函数里就挂好，早于任何 connect，避免竞态漏掉断线事件。
- **首连失败也重试**：设备可能刚好不在广播窗口，`connect()` catch 里也调 `_scheduleReconnect`。
- **Timer 可取消**：`_retryTimer` 在 `connect`/`disconnect`/`dispose`/关开关时都取消，防止「用户已经手动接管了，后台还偷偷重连」。

## 四、本课最容易漏的坑：重连后要重建订阅

这是面试高频、实战血泪点。**重连成功 ≠ 恢复原状**：

- 断线时旧的 GATT 句柄表作废（第 4 课）→ 重连后必须重新 `discoverServices`。
- 断线时 CCCD 被复位（第 5 课）→ 重连后**之前的通知订阅全部失效，必须重新订阅**。

漏掉第二条的症状极具迷惑性：「重连显示成功，设备也连上了，但再也收不到数据了」——因为没人重新往 CCCD 写 `01 00`。很多线上 Bug 就卡在这。

我们的解法：用 `_desiredNotify` 这个 Set 记录「用户想要哪些订阅」（`toggleNotify` 开时加入、关时移除），它**不随断线清空**。每次 (重)连成功后 `_resubscribeDesired()` 照这份清单把订阅重建一遍。把「订阅意图」（持久）和「订阅句柄」（随连接）分开，是这段设计的关键——又一次「意图状态 vs 链路状态」分层，和第 3 课 busy/connectionState 一脉相承。

## 五、推翻第 3 课的「离开页面即断开」

第 3 课 dispose 里无脑 `disconnect()`，注释就写着「第 7 课会调整」。现在调整：dispose 里先置 `_userInitiated = true` 再断开，让页面关闭走「主动断开」路径、不触发重连。真正的「App 退后台仍保持连接」属于后台保活，是第 9 课的内容——本课先把「页面内的意外掉线」这条主线做扎实。

## 六、动手任务

1. 连接 LightBlue 心率外设，订阅 2A37，看到曲线滚动。
2. **意外掉线实验**：保持 App 前台，直接在 LightBlue 关掉虚拟外设 → 观察状态卡片变成「断线，Xs 后第 N 次重连」，退避时间逐次变长。
3. **自愈实验**：在放弃前（1 分钟内）重新打开 LightBlue 外设 → App 自动重连成功，且**心率曲线自动恢复滚动**（证明订阅被自动重建，没让你手动再点订阅）。
4. **主动断开对照**：连接后点「断开」→ 确认它停在「未连接」、绝不自动重连。
5. **放弃实验**：关掉外设一直不开，看 6 次重试后进入「已放弃」，手动点连接可重新开始。
6. 关掉「意外掉线自动重连」开关，重复实验 2，确认掉线后不再重连。
