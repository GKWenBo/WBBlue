# 第 11 课（可选）：面试冲刺（1 课时）

> 前 10 课的知识，这课压缩成面试问答形态。左边是问题，右边是「能拿分的答法」。每题都能在前面课里找到实操依据——答的时候带一句「我在项目里…」，可信度翻倍。

## 一、基础与生命周期

**Q：讲一次 BLE 连接的完整生命周期。**
外设在 37/38/39 广播 → 主机扫描发现 → 发起连接、外设停播 → 连接参数生效（连接间隔/从机延迟/监督超时）→ MTU 协商 → 服务发现建句柄表 → 读写订阅 → 断开或超时。（第 1 课）

**Q：GAP 和 GATT 的区别？**
GAP 管「连接怎么建立、谁是主谁是从」；GATT 管「连上之后数据怎么组织」（Service→Characteristic→Descriptor 三层树）。谁存数据谁是 GATT Server，与谁发起连接无关。（第 1/4 课）

**Q：为什么连接后必须服务发现，能不能缓存句柄？**
句柄是 ATT 属性表的行号，随固件升级重排，不能跨连接复用。系统层 GATT 缓存在设备 OTA 后会失效，导致读写莫名失败——安卓靠 `refresh()` 反射清缓存，iOS 靠外设发 Service Changed。（第 4 课）

## 二、数据面四象限（最能体现深度）

**Q：写有几种？通知有几种？怎么选？**
四象限：写方向 Write Request（有确认、可靠、慢）vs Write Command（无确认、快）；通知方向 Indicate（有确认）vs Notify（无确认）。判据一句话：**丢了能自愈用快的，不能自愈用可靠的**。指令用 Write Request，OTA 大数据用 Write Command，流数据（心率）用 Notify，关键事件（Service Changed）用 Indicate。（第 4/5 课）

**Q：Write without response 是不是像 UDP，会丢包？**
不是。BLE 链路层有 CRC + 重传，空中不易丢。真正的风险是**接收端应用层缓冲溢出**（发太快对方处理不过来），所以高吞吐写要自己做应用层流控。（第 4 课）

**Q：`setNotifyValue(true)` 底层做了什么？**
两步：本地向系统栈注册回调 + 向 CCCD（0x2902）写 `01 00`。安卓原生经典 bug 是只做第一步忘了写 CCCD，结果一个通知都收不到。iOS 一步搞定且不暴露 CCCD。（第 5 课）

## 三、私有协议（企业核心，必被追问）

**Q：链路层有 CRC 了，为什么还要应用层协议和 CRC？**
BLE 特征是字节管道不是消息通道，链路 CRC 只保每个空中包，管不到「一条业务消息」的边界。粘包（固件合并发）、半包（超 MTU 切分）都要应用层用「帧头+长度+CRC」自己重建边界。（第 6 课）

**Q：你的帧怎么设计的？**
帧头 `A5 5A`（位模式交替、抗误同步）+ cmd + seq（请求响应配对）+ len（2 字节小端，带上限防坏帧撑爆内存）+ payload + CRC16（覆盖 cmd..payload）。（第 6 课）

**Q：收到半包/粘包/坏帧怎么处理？**
一个组包状态机：缓冲区累积，找帧头→读长度→等整帧→校验 CRC→吐帧。坏帧不整段丢，只丢帧头两字节逐字节重同步，因为坏帧后半段可能藏着下一个真帧头。（第 6 课，13 条单测覆盖）

**Q：OTA 固件升级怎么设计？**
Write Command 冲吞吐传固件块 + 应用层流控（每 N 包等一个进度 Notify）+ 分块 CRC + 断点重传。传输走快通道，可靠性在协议层自己挣。iOS 别忘了 Service Changed，否则升级后句柄对不上。（第 4/6 课）

## 四、稳定性（区分资深）

**Q：断线了怎么办？**
先区分主动断开 vs 意外掉线，只对后者重连（用 `_userInitiated` 标志）。重连用指数退避 + 抖动（避免射频拥塞和惊群），封顶放弃交给用户。**重连成功后必须重新发现服务 + 重新订阅**——CCCD 随连接复位，漏了就是「连上了但收不到数据」。（第 7 课）

**Q：安卓 status 133 是什么，怎么治？**
GATT_ERROR，安卓的「万金油错误」，射频冲突/协议栈状态错乱/缓存不符都归它。第一条军规：connect 前 stopScan。再配指数退避重试、断开后延迟再连、必要时清 GATT 缓存。把它当概率问题（压低发生率 + 状态机保证可恢复），不是逻辑问题。（第 3/7 课）

## 五、双端差异（iOS ↔ Android）

**Q：iOS 和 Android 蓝牙最大的区别？**
① 设备标识：iOS 给随机 UUID 拿不到 MAC，Android 给真 MAC → 跨端绑定要用广播里的序列号而非地址；② CCCD：iOS 一步、不可见，Android 两步；③ MTU：iOS 系统定、Android 可请求；④ 后台：iOS 系统代管 + State Restoration，Android 要前台服务。（第 2/5/9 课）

**Q：后台怎么保持连接？**
iOS：`bluetooth-central` 后台模式（退后台不断连、事件唤醒 10s）+ State Restoration（被回收后设备事件拉活），但用户上滑杀 App 无解。Android：前台服务（常驻通知）+ 电池优化白名单引导（国产 ROM 会额外杀）。两端都靠自动重连兜底。（第 9 课）

## 六、进阶专题（问到算你赚）

**Q：配对（Pairing）和绑定（Bonding）的区别？**
配对是本次连接的加密协商；绑定是把配对生成的密钥**长期保存**，下次连接直接加密不用重配。涉及加密特征（如带锁的写）时系统自动触发。安卓 `createBond`，iOS 由系统在访问加密特征时自动弹窗，App 无法直接发起。

**Q：iBeacon 是什么？和普通广播的关系？**
iBeacon 是苹果定义的一种广播格式（放在厂商数据 0xFF 里：UUID + Major + Minor + 发射功率），只广播不连接，用于室内定位/触发。iOS 用 CoreLocation 而非 CoreBluetooth 处理它——这也是第 2 课「安卓扫描要定位权限」的根源。

**Q：BLE Mesh 了解吗？**
Mesh 是多对多组网（灯控、楼宇），基于 BLE 广播的 managed flooding 转发，和我们做的「一对一 GATT 连接」是两套模型。面试知道「Mesh ≠ 多个 GATT 连接，是广播泛洪组网」即可，深入是专门方向。

**Q：连接间隔怎么权衡？**
小间隔 = 低延迟高吞吐但费电；大间隔 = 省电但响应慢。iOS 对间隔有严格限制（会拒绝过小的请求）。产品按场景定：OTA/实时控制期临时调小，日常保连调大。（第 1 课）
