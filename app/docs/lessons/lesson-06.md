# 第 6 课：私有二进制协议 ★企业核心（1 课时）

> 前 5 课学的是「蓝牙怎么用」，从这课起是「企业怎么用蓝牙」。真实产品里 App 和设备之间跑的不是裸字节，而是一套厂商自定义的帧协议。本课我们自己设计并实现一套，全程 TDD。

## 一、为什么需要「应用层协议」——链路层不是有 CRC 吗？

会被问的第一个问题。BLE 链路层确实有 CRC 和重传，但它只保证**「每个空中包」**的完整，管不到**「你的一条业务消息」**。原因是 BLE 特征本质是一根**字节管道**，不是消息通道：

- **粘包**：外设固件缓冲区攒了两帧一起 `notify` 出来，到达时 App 看到的是一串连续字节，边界丢了。
- **半包**：一帧超过 MTU-3，被切成多个通知分次到达。
- **无边界**：GATT 只定义「这次收到 N 个字节」，从不告诉你「这是第几条消息、到哪结束」。

所以应用层必须自带**帧同步**：用固定帧头找起点、用长度字段定终点、用 CRC 校验判断边界猜得对不对。这套东西就是私有协议。

## 二、帧结构设计：每个字段都不可省

我们设计的帧（多字节字段一律小端）：

```
┌──────┬──────┬─────┬─────┬────────┬─────────┬────────┐
│ 0xA5 │ 0x5A │ cmd │ seq │ len(2) │ payload │ crc(2) │
└──────┴──────┴─────┴─────┴────────┴─────────┴────────┘
                └──────────  CRC 覆盖范围  ─────┘
```

| 字段 | 作用 | 省了会怎样 |
|---|---|---|
| 帧头 `A5 5A` | 在字节流里定位帧起点 | 粘包/垃圾字节后再也找不到帧边界 |
| `cmd` | 命令字，区分业务（读配置/写配置/OTA…） | 收到帧不知道是什么请求 |
| `seq` | 序号，请求响应配对 + 丢包检测 | 并发请求时对不上是哪条的回复 |
| `len` | 负载长度，定位帧尾 | 不知道 payload 多长，无法切分粘包 |
| `payload` | 业务数据 | — |
| `crc` | 覆盖 cmd..payload，校验完整性 | 无法判断帧边界是否猜对、有无误码 |

两个防御性设计（面试加分点）：

- **帧头选 `A5 5A`**：`0xA5` = `10100101`，`0x5A` = `01011010`，位模式交替、互为反码，在随机字节流里碰巧出现的概率低，抗误同步。
- **`len` 上限（512）**：损坏的 len 字段可能是 `0xFFFF`，若无条件等 65535 字节会让接收缓冲无限膨胀。超上限直接判坏帧丢弃——**永远不要相信长度字段**。

## 三、CRC-16/CCITT-FALSE

选它是因为它是嵌入式设备最常见的 CRC 变体（Modbus、蓝牙 SIG 很多规范都用），固件同事拿到「CRC-16/CCITT-FALSE, poly 0x1021, init 0xFFFF」这一句就能对上。实现见 [packet.dart](../../lib/core/protocol/packet.dart) 的 `crc16`——逐字节、逐位异或移位，纯整数运算。单测里用了业界标准校验值：字符串 `"123456789"` 的 CRC 必须等于 `0x29B1`，这是验证任何 CRC 实现对不对的「金标准」。

## 四、组包状态机：半包/粘包/坏帧的统一处理器

核心是 `PacketAssembler`（[packet.dart](../../lib/core/protocol/packet.dart)），它持有一个跨多次调用累积的缓冲区，每次 `feed` 新字节后循环尝试解帧：

1. **找帧头**，丢弃头之前的垃圾字节（处理垃圾前缀）；
2. 不足 6 字节（读不出 len）就**等下次**（处理半包）；
3. 读出 len，超上限→报错并跳过帧头两字节**重新同步**；
4. 整帧未到齐就等（处理半包）；
5. CRC 校验，错→报错并**从帧头之后逐字节重同步**（一个坏帧不拖垮整条流）；
6. 校验通过→吐出 `Packet`，缓冲区移除该帧，`while` 继续解下一帧（处理粘包）。

「坏帧后逐字节重同步」是这套设计的灵魂：坏帧的后半段里可能藏着下一个真帧头，所以不能整段丢，只丢当前帧头的两个字节，让 `_indexOfHeader` 从后面重新找。测试 `CRC 错帧被报错并重同步，后续好帧仍可解出` 守护这条。

## 五、分包：把大数据按 MTU 切开

`PacketCodec.chunks(frame, mtuPayload)` 把编码后的整帧按写负载上限切块。上限 = **ATT_MTU - 3**（3 是 ATT 头开销），代码里从 `device.mtuNow` 取。这就是第 3 课「连接后自动请求 MTU 512」和第 4 课 OTA 流控的收口：MTU 越大，单包能塞的越多，分包数越少、吞吐越高。

发送端流程（见 [protocol_session.dart](../../lib/features/protocol/protocol_session.dart) 的 `send`）：`Packet → encode → chunks → 逐块 write`。接收端 `PacketAssembler` 自动把这些块拼回完整帧——**分包在发送端、组包在接收端，两者独立**。

协议层是**纯 Dart、零蓝牙依赖**，13 条单测不碰硬件就跑完——这不是洁癖，是第 8 课「可测试架构」的地基：业务逻辑与传输解耦，才能 Mock、才能 CI。

## 七、动手任务

1. 安卓 nRF Connect → Configure GATT server → 建一个私有 Service（128-bit UUID），下挂两条特征：一条 **Write**（下发）、一条 **Notify**（上报）。开播。
2. iPhone 跑 App → 连接 → 进「私有协议控制台」→ 绑定这两条特征。
3. 在控制台发一帧：cmd 填 `10`，payload 填 `01 02 03` → 观察日志里 App 打出「↑ 帧 … 共 N 包」。回 nRF Connect 的 SERVER 页看写特征收到的原始字节，对照帧结构逐字节认一遍（`A5 5A 10 00 03 00 01 02 03` + 2 字节 CRC）。
4. 反向验证组包：在 nRF Connect 里让 Notify 特征发送一段你手工拼的帧字节（可故意分两次发验证半包，或一次发两帧验证粘包），看控制台是否正确解出。故意改错 1 个字节，看是否报「坏帧」并在下一帧恢复。

> 没有真机固件也能验证核心逻辑：`flutter test test/packet_codec_test.dart` 覆盖了全部边界。控制台是让你在真实字节管道上「亲眼看见」半包粘包。
