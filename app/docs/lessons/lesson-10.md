# 第 10 课：综合项目「设备管家」全流程验收（1 课时）

> 不引入新知识，把前 9 课串成一条完整的产品级链路，做一次双端真机验收。App 现在有了正式首页 [home_page.dart](../../lib/features/home/home_page.dart)，三个入口：真机模式、离线演示、双端信息。这一课是给你一份能对着做、也能对着讲的验收脚本。

## 一、架构总览：一张图讲清全貌

```
                    ┌─────────────── UI 层 ───────────────┐
   HomePage ──▶ ScanPage ──▶ DevicePage ──▶ ProtocolConsole
   （门面）      （扫描）      （连接/GATT/订阅/重连）   （私有协议）
                    │            MockDemoPage（离线演示）
                    ▼
   ┌──────────── Controller 层（ChangeNotifier）────────────┐
   ScanController      DeviceController      ProtocolSession
   （扫描状态）     （连接/重连状态机/订阅）   （帧收发）
                    │
                    ▼
   ┌──────────── Core 层（纯逻辑，全单测覆盖）──────────────┐
   BleCentral 抽象接口 ── RealBleCentral / MockBleCentral
   protocol/packet     heart_rate     hex     backoff
```

三条设计主线，面试时能画出这张图 + 说清这三点，就是及格线以上：

1. **单向依赖**：UI → Controller → Core，永不反向。
2. **纯逻辑下沉**：协议、解析、退避全是纯函数，40 条单测不碰硬件。
3. **抽象隔离硬件**：`BleCentral` 接口把 FBP 关进防腐层，可 Mock 可测可换。

## 二、贯穿全程的三组「分层状态」

这是整个课程反复强调的思想，验收时重点自检：

| 意图/UI 状态（自己维护） | 链路/协议状态（来自底层） | 出自 |
|---|---|---|
| `busy`（动作进行中） | `connectionState`（通断） | 第 3 课 |
| `_desiredNotify`（想订阅什么） | `isNotifying`（CCCD 实况） | 第 5/7 课 |
| `_userInitiated`（是不是我要断的） | `disconnectReason`（为什么断） | 第 7 课 |

混用左右两列，就是蓝牙 App 状态错乱的总根源。

## 三、双端真机验收脚本（逐条打勾）

### A. 扫描与连接（第 2-3 课）
- [ ] 首页进「真机模式」→ 开扫描，Android 首次弹「附近的设备」权限
- [ ] 目标设备出现，RSSI 随距离变化；关掉外设 4 秒后条目自动消失（幽灵治理）
- [ ] 点进设备 → 连接成功，状态卡显示「已连接」；读取 RSSI 有值

### B. GATT 读写与订阅（第 4-5 课）
- [ ] 服务浏览器列出服务树，特征带属性徽标
- [ ] 对可读特征「读取」，HEX + 文本双显示
- [ ] 对可写特征「写入」`48 69`，对端看到值变
- [ ] 订阅心率（2A37）→ 顶部出现实时曲线，滚动刷新

### C. 私有协议（第 6 课）
- [ ] 进「私有协议控制台」→ 绑定写/通知特征
- [ ] 发送 cmd=10 payload=01 02 03 → 对端收到符合帧结构的字节
- [ ] 让对端回帧（含故意坏帧）→ 控制台解出好帧、坏帧报错并重同步

### D. 自动重连（第 7 课）
- [ ] 连接+订阅后关掉外设 → 状态变「断线，Xs 后第 N 次重连」，退避递增
- [ ] 1 分钟内重开外设 → 自动重连成功，**心率曲线自动续上**（订阅自动重建）
- [ ] 点「断开」→ 绝不自动重连

### E. 离线与双端（第 8-9 课）
- [ ] 首页「离线演示」→ 模拟器/真机零外设跑通心率曲线 + 协议回帧 + 故障注入
- [ ] 首页「双端差异」→ 正确识别当前平台，对照表可讲
- [ ] `flutter test` 全绿（40 条）

## 四、离线兜底：没带第二台手机也能验收

面试现场往往只有一台设备。用**离线演示**（首页第二个入口）可以零外设展示：扫描→连接→心率曲线→协议往返→故障注入下的重连与坏帧恢复，覆盖 B/C/D/E 的核心。加上 `flutter test` 的 40 条，足以证明全链路正确性。

## 五、动手任务

1. 按第三节 A-E 逐条在双端真机上打勾（有第二台设备走真机，没有就用离线演示 + 单测兜底）。
2. 对着第一节的架构图，自己复述一遍数据从 UI 到 Core 的流向。
3. 把 D 组的「掉线→自动重连→曲线续上」录一小段——这是作品集/面试演示的最佳 30 秒。

## 验收

1. A-E 五组清单在真机（或离线兜底）下全部通过。
2. 能对着架构图讲清三条主线（单向依赖 / 纯逻辑下沉 / 抽象隔离硬件）。
3. 回答：① 项目里有哪三组「意图状态 vs 链路状态」的分层？混用会怎样？② 只有一台手机，你怎么完整演示这个 App 的能力？③ 如果要把 flutter_blue_plus 换成别的库，需要改动哪些文件？为什么只有这些？
